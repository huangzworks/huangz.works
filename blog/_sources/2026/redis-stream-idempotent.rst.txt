Redis 8.6版本流的幂等生产机制新特性介绍
=======================================

.. tip::

    观看本文的视频版本：https://www.bilibili.com/video/BV1NVNuzPEQ1/

大家好，我是黄健宏。继上次给大家介绍完Redis
8.6新引入的\ `热键追踪特性 <https://huangz.works/blog/2026/redis-hotkeys.html>`__\ 之后，今天我继续来给大家介绍Redis
8.6新引入的另一个特性——流的幂等生产机制。

在Redis
8.6之前，如果生产者在调用\ ``XADD``\ 命令之后、收到回复之前断连了，那么它将无法确定该消息是否已成功发送，更无法将该消息判定为已递送。

在这种情况下，为了确保消息被添加至流中，生产者必须使用相同的消息重新调用\ ``XADD``\ 。但是如果这种生产操作不具备幂等性，那么重新发送可能会导致消息被重复投递，而采用幂等消息处理机制可以保证消息在上述断连情况下最多也只投递一次。

--------------

.. image:: images/stream-case-1.jpg

图1：重复投递的情况（图片引用自Redis官方博客，下同）

--------------

为了实现“同一条消息最多只投递一次”的幂等性保证，Redis
8.6引入了幂等ID参数，生产者可以在向流中添加每条消息时关联一个唯一且幂等的ID：

::

   XADD key … [<IDMP pid iid> | <IDMPAUTO pid>] … <*|id> field value [field value …]

其中\ ``IDMP pid iid``\ 选项要求生产者显式地提供生产者标识符（pid）和幂等标识符（iid），而\ ``IDMPAUTO pid``\ 选项则只要求生产者提供自己的生产者标识符，而幂等标识符则根据传入的消息内容自动生成。

   **注意**

   由于\ ``IDMPAUTO``\ 选项会根据消息内容自动生成幂等标识符，所以使用这一选项的\ ``XADD``\ 命令无法发送内容重复的消息：因为命令会以为你想要重复发送之前已经发送过的消息，并为它生成之前使用的幂等标识符，从而导致相同内容的消息重发失败。

与\ ``IDMP``\ 选项和\ ``IDMPAUTO``\ 选项相对应的是，服务器在执行带有这两个选项的\ ``XADD``\ 命令之后，会保存起一定数量的最近接收消息的\ ``(pid, iid)``\ 对，并在之后每次接收到带有这两个选项的\ ``XADD``\ 命令时，检查是否有已经存在相同的\ ``(pid, iid)``\ 对：如果是的话那么说明消息已经发送过，服务器就会把它忽略掉，这样一来就解决了同一条消息重复发送的问题。

--------------

.. image:: images/stream-case-2.jpg

图2：解决同一消息重复投递的方法

--------------

最后，为了控制被记录的\ ``(pid, iid)``\ 对数量，Redis
8.6还引入了新的设置命令：

.. code:: txt

   XCFGSET key [IDMP-DURATION seconds] [IDMP-MAXSIZE maxsize]

其中\ ``IDMP-DURATION``\ 用于设置每个iid的最长保存秒数，而\ ``IDMP-MAXSIZE``\ 则用于设置每个pid能够保存的最近iid数量。
